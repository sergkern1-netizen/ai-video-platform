# Надёжное достижение целевой длины сценария для long-видео — дизайн

Дата: 2026-06-18

## Проблема

`generate_script(topic, "long")` (`backend/pipeline/script_generator.py`) просит GPT написать ~1500 слов / минимум 45 сцен на 10-минутное видео, но модель систематически останавливается раньше добровольно (`finish_reason="stop"`, токены ещё были в запасе — не ограничение API). Замеры в этой сессии:

| Модель | Попытка | Слов | Сцен |
|---|---|---|---|
| gpt-4o-mini | 1 | ~290 (152с видео) | — |
| gpt-4o-mini | 2 | 527 | 35 |
| gpt-4o-mini (2-step промпт) | 3 | 367 | 19 |
| gpt-4o-mini (2-step промпт) | 4 | 352 | 16 |
| gpt-4o-mini (2-step промпт) | 5 | 311 | 14 |
| gpt-4o | 6 | 557 | 44 |

Усиление промпта (явный минимум, двухэтапная структура "сначала план — потом сцены") не дало стабильного улучшения. Смена модели на gpt-4o дала近 целевое число сцен (44 из 45), но не слов (557 из 1500).

**Важный нюанс:** реальная длительность итогового видео определяется TTS-озвучкой `script.body` (количеством слов), а не полем `Script.duration_sec` или суммой `Scene.duration_sec` — эти поля используются рендерером только как относительные веса между сценами (`backend/pipeline/video_renderer.py`, `_build_background`). `Script.duration_sec` не используется больше нигде в пайплайне (проверено `grep` по `backend/pipeline/` и `backend/tests/`, кроме собственных тестов `script_generator`), так что его финальное значение можно зафиксировать декларативно.

## Решение: итеративное доращивание (только для `format="long"`)

1. Первый вызов GPT (модель `gpt-4o`, как уже установлено) с текущим `_LONG_PROMPT` → парсим `scenes`, считаем `word_count = len(body.split())`.
2. Если `word_count < _LONG_TARGET_WORDS * _LONG_MIN_RATIO` (1500 × 0.9 = 1350) — делаем **продолжающий** вызов в том же диалоге:
   - добавляем исходный ответ модели как `assistant`-сообщение в историю,
   - добавляем `user`-сообщение с точным указанием текущего количества слов, недостающего количества, и просьбой написать **только новые** сцены по **новым, ещё не затронутым** аспектам темы (явно: не повторять и не пересказывать уже написанное).
3. Повторяем шаг 2 до достижения порога или после `_LONG_MAX_CONTINUATIONS = 3` продолжений (итого до 4 вызовов API на сценарий).
4. Если после всех попыток порог так и не достигнут — отдаём то, что накопилось (не считаем это ошибкой/не роняем pipeline; частичный сценарий лучше отсутствующего).
5. Финальный `Script`: `title`/`mood` — из первого ответа; `scenes` — конкатенация всех накопленных сцен по порядку поступления; `body` — склейка текста всех сцен через `" "`; `duration_sec` — фиксированно `600` (декларативное значение, как и раньше).

Для `format="short"` поведение не меняется (один вызов, `gpt-4o-mini`, текущий `_SHORT_PROMPT`).

## Компоненты

### `backend/pipeline/script_generator.py`

- Новые константы: `_LONG_TARGET_WORDS = 1500`, `_LONG_MIN_RATIO = 0.9`, `_LONG_MAX_CONTINUATIONS = 3`.
- Существующий retry-on-invalid-JSON (`max_retries`) остаётся как внутренний механизм одного вызова (если конкретный ответ — невалидный JSON, повторяем тот же вызов до `max_retries`, как сейчас) — он не заменяется, а оборачивается циклом доращивания снаружи.
- Новая функция `_continue_prompt(word_count: int, target: int) -> str`, возвращающая текст user-сообщения для продолжения, с подстановкой фактических чисел.
- `generate_script` для `format == "long"` после получения первого валидного `scenes`/`body` входит в цикл доращивания, накапливая `messages` (chat history) и `scenes` между итерациями.
- Парсинг продолжающего ответа: JSON-схема для продолжения — `{"scenes": [...]}` (без `title`/`mood`/`duration_sec` — они берутся из первого ответа). Невалидный JSON в продолжении — повторяем тот же continuation-вызов до `max_retries` (используем тот же внутренний механизм, что и для первого вызова), а если все попытки внутри одной continuation-итерации не дали валидного JSON — прерываем цикл доращивания и отдаём то, что уже накоплено (graceful degradation, без падения pipeline).

## Тестирование

Юнит-тесты (моки `OpenAI`, без реальных вызовов):
- Если первый ответ для `long` уже достиг порога (≥1350 слов) — `chat.completions.create` вызывается **один раз**, без продолжений.
- Если первый ответ короче порога — модуль делает продолжающий вызов; финальный `Script.body` содержит склейку текста из обоих ответов (порядок сохранён), `scenes` — объединённый список.
- Если после `_LONG_MAX_CONTINUATIONS` продолжений порог так и не достигнут — функция возвращает накопленный (неполный) результат, не бросает исключение.
- Невалидный JSON в продолжении — переиспользует retry-логику; если не восстановился, возвращает накопленный до этого момента результат.
- `format="short"` — поведение не меняется, существующие тесты для short остаются без изменений.

Ручная проверка (как и раньше для этого плана) — реальный вызов `generate_script(topic, "long")` и подсчёт фактических слов/сцен, без полного E2E рендера (дёшево и быстро для итерации).

## Вне скоупа

- Изменения для `format="short"`.
- Изменение того, как `duration_sec` влияет на рендер (остаётся как есть — относительный вес сцены).
- Параллелизация continuation-вызовов (выполняются строго последовательно, так как каждый зависит от истории предыдущего).
